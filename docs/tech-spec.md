## Problem & Solution

### Problem Statement

The daily summary system needs to produce structured, LLM-optimized markdown with machine-readable metadata, cross-day continuity tracking, and weekly synthesis. The system must be fully configurable without editing prompt internals, and easy to set up for any user.

### Proposed Solution

The daily summary system consists of:

1. **Structured format** — YAML frontmatter, semantic sections (Decisions & Rationale, Open Loops, Blockers, Next Steps), and cross-day context loading.
2. **Configuration file** — All user-specific variables and preferences in a `config.yaml`, generated by the setup wizard.
3. **Two-repo architecture** — A public repo (skills, docs, setup) and a private repo (generated summaries, user config).
4. **Interactive setup wizard** — Guided onboarding that configures everything.

---

## Requirements

### What Needs to Be Built

- **R1: YAML frontmatter on every summary file** — Machine-readable metadata block containing: date, weekday, tags (auto-derived), data sources used, and quality flag (`complete`, `partial`, `manual-only`).
- **R2: Semantic sections** — Summary bullets, Decisions & Rationale, Open Loops, Blockers, Next Steps, Transcript Source. Each optional (omitted if empty).
- **R3: Cross-day context loading** — Before writing today's summary, read previous day's summary to carry forward unresolved Open Loops and pending Next Steps.
- **R4: Weekly rollup task** — Runs on configured day, reads the week's daily summaries, produces a `weekly-rollup-<YYYY-Www>.md`.
- **R5: Configuration file (`config.yaml`)** — All user-specific variables and output preferences in a single YAML file, generated by the setup wizard.
- **R6: Interactive setup wizard** — Claude Code skill that guides the user through configuration, creates scheduled tasks, and sets up the private summaries repo.
- **R7: Skill prompts** — Skills read from `config.yaml`, produce the structured format, and load cross-day context.

### What This Does NOT Include

- A web UI, mobile app, or dashboard for viewing summaries
- Real-time / intra-day logging or notifications
- Team-level summaries or multi-user support
- Migration of historical summaries from other tools

## Technical Approach

### Technology Stack

- **Runtime:** Claude Code scheduled task (local execution)
- **Language/Format:** Markdown files with YAML frontmatter
- **Storage:** Private Git repository on GitHub (created during setup)
- **Data Sources:** MCP connectors — Google Calendar, Gmail, Slack; Google Drive for Desktop (local file scanning)
- **Configuration:** YAML (`config.yaml` in the private summaries repo)
- **Orchestration:** Claude Code skill prompts (`.claude/skills/`)

### Architecture Overview

```
┌─────────────────────────────────────────────────┐
│           Claude Code Scheduled Task            │
│           (user-configured schedule)            │
├─────────────────────────────────────────────────┤
│                                                 │
│  1. Read config.yaml (variables & preferences)  │
│  2. Read yesterday's summary (cross-day ctx)    │
│  3. Query MCP connectors (Calendar, Gmail,      │
│     Slack) + scan local Drive files             │
│  4. Compose narrative → structured summary      │
│  5. Write .md file → git commit → git push      │
│                                                 │
├─────────────────────────────────────────────────┤
│  Inputs:          │  Output:                    │
│  - config.yaml    │  - <slug>-daily-summary-    │
│  - MCP data       │    <YYYY-MM-DD>.md          │
│  - Yesterday's    │  - Git commit + push        │
│    summary        │                             │
└─────────────────────────────────────────────────┘
```

**Weekly rollup** follows the same pattern but reads daily files instead of MCP sources.

### Data Model

**Summary file structure:**

```yaml
# --- YAML frontmatter ---
---
date: "2026-03-24"
weekday: "Tuesday"
author: "Jane Smith"
quality: "complete"          # complete | partial | manual-only
sources_used:
  - google-calendar
  - gmail
  - slack
  - google-drive
sources_empty: []            # list any that returned no data
open_loops_carried: 2        # count from yesterday
tags:
  - hiring
  - compliance
  - product-launch
---
```

**Markdown body sections (in order):**

```markdown
# Tuesday - 2026-03-24

## Summary
- <bullet 1>
- <bullet n>

## Decisions & Rationale
- <decision>: <why>

## Open Loops
- <thing that is unresolved, with context>

## Blockers
- <thing blocking progress>

## Next Steps
- <planned action for tomorrow or near-term>

## Transcript Source (Cleaned)
<natural-language narrative>
```

### Config file structure (`config.yaml`):

```yaml
# ============================================================
# Daily Summaries — Configuration
# Generated by the setup wizard. Re-run setup to modify.
# ============================================================

config_version: 1

# --- Your identity ---
user:
  full_name: "Jane Smith"
  timezone: "America/New_York"
  file_slug: "jane-smith"
  slack_user_id: "U0XXXXXXXXX"

# --- Google Drive (optional — for local Drive file tracking) ---
google_drive:
  artifacts_folder_path: "/path/to/My Drive/work-artifacts"

# --- Schedule ---
schedule:
  days: [1, 2, 3, 4, 5]       # 0=Sun, 1=Mon, ..., 6=Sat
  time: "17:00"                 # 24h format, local time
  rollup_enabled: true
  rollup_day: "friday"

# --- Summary format preferences ---
format:
  sections:
    summary: true
    decisions_and_rationale: true
    open_loops: true
    blockers: true
    next_steps: true
    transcript_source: true
  bullet_target: "5-15"
  yaml_frontmatter: true

# --- Cross-day context ---
context:
  load_previous_day: true
  lookback_days: 2

# --- Data sources ---
data_sources:
  google_calendar:
    enabled: true
  gmail:
    enabled: true
    action_only: true
  slack:
    enabled: true
    sent_only: true
  google_drive:
    enabled: true
```

### API Design

Not applicable — no APIs exposed. The system is a scheduled batch job that reads from MCP connectors and writes to the filesystem/git.

## Non-Functional Requirements

### Performance

- Daily task completes within 5 minutes (MCP queries + summary generation + git push).
- Weekly rollup completes within 3 minutes (reading local files + synthesis).

### Security

- No secrets stored in the public repo.
- Private summaries repo contains only the user's own work activity.
- MCP auth is handled by Claude Code's connector layer.
- `config.yaml` lives in the private repo, not the public one.

### Other

- Summary files must be valid markdown that renders correctly on GitHub.
- YAML frontmatter must be valid YAML parseable by standard libraries.
- Filenames: `<slug>-daily-summary-<YYYY-MM-DD>.md` and `weekly-rollup-<YYYY-Www>.md`.

## Dependencies

- Claude Code with scheduled task support and MCP connectors.
- MCP connectors for Google Calendar, Gmail, and Slack.
- Google Drive for Desktop (optional, for local file scanning).
- GitHub account with ability to create repositories.
- Git configured on the local machine.

## Risks & Mitigation

- **Risk:** Cross-day context loading creates stale content (open loops repeated forever)
  - **Mitigation:** Items carried forward get a `(carried from <date>)` tag; items older than 5 days auto-dropped.

- **Risk:** `config.yaml` parsing fails silently
  - **Mitigation:** Skill prompt includes error message instructing user to run setup if config is missing/invalid.

- **Risk:** Weekly rollup misses a day (summary wasn't generated)
  - **Mitigation:** Rollup notes which days are present/missing and proceeds with available data.

- **Risk:** MCP connector UUIDs differ per user account
  - **Mitigation:** Skills use keyword-based ToolSearch to discover tools dynamically rather than hardcoding UUIDs.
